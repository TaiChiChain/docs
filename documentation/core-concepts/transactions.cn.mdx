---
title: 'Transactions'
---

交易是来自账户的加密签名指令。账户将发起交易以更新 Axiom 网络的状态。最简单的交易是将 Axiom 从一个账户转移到另一个账户。

改变链状态的交易需要广播到整个网络。任何节点都可以广播交易请求。验证节点收到交易后，会执行该交易并将其包含在一个区块中，然后将区块传播到整个网络。

只有在验证节点将交易打包到一个区块中，并且该区块得到确认后，交易才最终确认。

交易包括以下信息：

from - 发送者地址，该地址将对交易进行签名。这应该是一个 EOA，因为合约账户不能发送交易。
recipient - 接收者地址（如果地址是 EOA，交易将转移价值。如果地址是合约地址，交易将执行合约代码）。
signature - 发送者的身份标识。当发送者使用私钥发送交易时，会生成签名。
nonce - 有序递增的计数器，表示来自账户的交易数量。
value - 从发送者转移到接收者的 AXO 数量。
data - 可选字段，能够包含任意数据。
gasLimit - 交易可以消耗的最大数量的燃料单位，指定每个可计算步骤所需的燃料单位。
交易对象如下所示：

```
{
  from: "0x8219e5E23396587Bf1969eFd1B211F379aa29494",
  to: "0x44D848b5DC373dafDde447c158876579C50c34dF",
  gasLimit: "21000",
  maxFeePerGas: "300"
  maxPriorityFeePerGas: "10"
  nonce: "0",
  value: "10000000000",
}
```

交易需要使用发送者的私钥进行签名。这证明交易只可能来自发送者，而不是从作恶者发出。

Axiom 与像 Geth 这样的以太坊客户端兼容，可以通过与 Geth 兼容的以太坊客户端来处理签名过程。

示例 JSON-RPC 调用：

```
{
  "id": 2,
  "jsonrpc": "2.0",
  "method": "account_signTransaction",
  "params": [
    {
      "from": "0x1923f626bb8dc025849e00f99c25fe2b2f7fb0db",
      "gas": "0x55555",
      "maxFeePerGas": "0x1234",
      "maxPriorityFeePerGas": "0x1234",
      "input": "0xabcd",
      "nonce": "0x0",
      "to": "0x07a565b7ed7d7a678680a4c162885bedbb695fe0",
      "value": "0x1234"
    }
  ]
}
```

# 交易的类型

Axiom 拥有四种不同类型的交易：

- 常规交易（Regular Transactions）：从一个账户发送到另一个账户的交易。这种类型的交易涉及将 AXM 从一个地址转移到另一个地址，并可以包括一个可选的数据字段以提供额外信息。

- 系统执行交易（System Execution Transaction）：这些交易用于与 Axiom 网络原生集成的系统合约进行交互。在这种情况下，交易的“to”地址是系统合约的地址，交易数据字段包含要在合约内执行的函数调用和参数。

- 合约部署交易（Contract Deployment Transactions）：这些交易没有“to”地址；相反，它们包括用于部署智能合约的代码和初始化参数。在这种类型的交易中，交易数据字段包含合约的字节码和构造函数参数。

-合约执行交易（Contract Execution Transactions）：这些交易用于与已部署在 Axiom 网络上的智能合约进行交互。在这种情况下，交易的“to”地址是智能合约的地址，交易数据字段包含要在合约内执行的函数调用和参数。

这些不同类型的交易允许在 Axiom 网络上进行资金转移、部署智能合约，以及与智能合约和系统合约进行交互。
